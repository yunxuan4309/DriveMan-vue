# 排班管理与车辆管理 — 前端对接接口文档

---

## 一、功能概述

### 1.1 业务背景

驾校的教练车和训练场地是有限的共享资源。每位教练在上课前需要先申请车辆和场地，管理员审批通过后，该时段才会开放给学员约课。

### 1.2 核心业务流程

```
① 管理员录入车辆信息           → 车队管理
② 教练提交排班申请(车+场地+时段) → 系统自动冲突检测
③ 管理员审核排班(通过/拒绝)     → 资源分配确认
④ 学员查看自己教练的可约时段    → 自动过滤：仅已通过 + 名额未满 + 未来时间
⑤ 学员约课(必须在教练排班内)    → 系统校验归属+时段+名额
⑥ 教练确认/拒绝/完成约课       → 约课生命周期管理
```

### 1.3 涉及的数据表

| 表名 | 说明 | 本次变更类型 |
|------|------|------------|
| `vehicle` | 教练车车队 | **新增** |
| `coach_schedule` | 教练排班/车辆使用申请 | **新增** |
| `venue` | 场地（新增 max_vehicles / supported_types 字段） | 扩展 |
| `appointment` | 约课（新增 schedule_id 字段，状态增加 4=已完成） | 扩展 |

### 1.4 通用约定

- **基础路径**: `http://localhost:9500`
- **认证方式**: 请求头携带 `Authorization: Bearer <token>`
- **统一响应**: `{ state: 20000, message: null, data: ... }`
- **分页响应**: `{ records: [...], total: N, pages: N, current: N, size: N }`
- **角色**: `1=学员` / `2=教练` / `3=管理员`

## 二、车辆管理（管理员专用）

### 2.1 分页查询车辆列表

```
GET /vehicles?page=1&size=10&vehicleType=C1&status=1
```

**请求参数：**

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | int | 否 | 页码，默认 1 |
| size | int | 否 | 每页条数，默认 10 |
| vehicleType | String | 否 | 筛选车型：C1 / C2 / B1 / N1 / N2 / N3 |
| status | int | 否 | 筛选状态：1=空闲 2=使用中 3=维修 4=报废 |

**响应示例：**

```json
{
  "state": 20000,
  "data": {
    "records": [
      {
        "id": 1,
        "plateNumber": "渝A·C1001",
        "vehicleType": "C1",
        "brand": "大众",
        "model": "桑塔纳",
        "seats": 5,
        "status": 1,
        "remarks": "手动挡教练车",
        "createTime": "2026-06-05T12:00:00"
      }
    ],
    "total": 6,
    "pages": 1,
    "current": 1,
    "size": 10
  }
}
```

**status 字段枚举：**

| 值 | 含义 | 前端展示建议 |
|----|------|------------|
| 1 | 空闲 | 绿色标签 |
| 2 | 使用中 | 蓝色标签 |
| 3 | 维修 | 橙色标签 |
| 4 | 报废 | 灰色标签（不可选） |

---

### 2.2 查询可用车辆（供教练排班时选择）

```
GET /vehicles/available?vehicleType=C1
```

> 仅返回 status=1 (空闲) 的车辆。教练提交排班申请时，前端应调用此接口获取可选车辆列表。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| vehicleType | String | 否 | 按车型筛选，不传则返回全部可用车辆 |

---

### 2.3 新增车辆

```
POST /vehicles
Content-Type: application/json

{
  "plateNumber": "渝A·C1003",
  "vehicleType": "C1",
  "brand": "大众",
  "model": "捷达",
  "seats": 5,
  "status": 1,
  "remarks": "新购入手动挡教练车"
}
```

**请求体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| plateNumber | String | **是** | 车牌号，必须唯一 |
| vehicleType | String | **是** | 车型 |
| brand | String | 否 | 品牌 |
| model | String | 否 | 型号 |
| seats | int | 否 | 座位数，默认 5 |
| status | int | 否 | 状态，默认 1 |
| remarks | String | 否 | 备注 |

---

### 2.4 修改车辆

```
PUT /vehicles/{id}
Content-Type: application/json

{
  "status": 3,
  "remarks": "离合器故障送修"
}
```

> 只需传需要修改的字段，未传字段保持不变。

---

### 2.5 删除车辆

```
DELETE /vehicles/{id}
```

> 逻辑删除，不会物理删除数据库记录。已被排班引用的车辆不建议删除。

## 三、排班管理

### 3.1 排班状态机

```
  教练提交
    │
    ▼
┌─────────┐   管理员通过   ┌─────────┐   时段结束   ┌─────────┐
│ 0-待审核 │ ────────────→ │ 1-已通过 │ ──────────→ │ 3-已完成 │
└─────────┘               └─────────┘             └─────────┘
    │                         │
    │ 管理员拒绝               │ 教练取消
    ▼                         ▼
┌─────────┐               ┌─────────┐
│ 2-已拒绝 │               │ 4-已取消 │
└─────────┘               └─────────┘
```

| status | 含义 | 可执行操作 |
|--------|------|-----------|
| 0 | 待审核 | 管理员可审批 / 教练可取消 |
| 1 | 已通过 | 学员可约课 / 教练可取消 |
| 2 | 已拒绝 | 终态，不可操作 |
| 3 | 已完成 | 终态，时段已过 |
| 4 | 已取消 | 终态，教练主动取消 |

---

### 3.2 教练端 — 提交排班申请

```
POST /coach-portal/schedules
Content-Type: application/json

{
  "vehicleId": 1,
  "venueId": 4,
  "licenseType": "C1",
  "startTime": "2026-06-10T09:00:00",
  "endTime": "2026-06-10T11:00:00",
  "maxStudents": 1,
  "applyReason": "周三上午常规教学"
}
```

**请求体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| vehicleId | int | **是** | 车辆ID（应先调用 `GET /vehicles/available` 获取可选车辆） |
| venueId | int | **是** | 训练场地ID（应先调用 `GET /venues?venueType=2` 获取训练场地列表） |
| licenseType | String | **是** | 培训车型，必须与车辆车型一致且教练准教车型包含此车型 |
| startTime | datetime | **是** | 开始时间，格式 `yyyy-MM-ddTHH:mm:ss`，必须在未来 |
| endTime | datetime | **是** | 结束时间，必须晚于 startTime |
| maxStudents | int | 否 | 该时段最多可容纳学员数，默认 1（一对一教学） |
| applyReason | String | 否 | 申请说明 |

**系统自动校验（7 项），任一不通过则返回错误：**

| # | 校验项 | 错误提示（state=4xxxx） |
|---|--------|----------------------|
| 1 | 时间合法性 | "开始时间必须早于结束时间" / "排班开始时间必须在未来" |
| 2 | 教练车型匹配 | "您的准教车型(C1,C2)不包含B1" |
| 3 | 车辆车型匹配 | "该车辆车型为C2，不支持C1培训" |
| 4 | 车辆可用性 | "该车辆当前不可用（状态：维修）" |
| 5 | 场地类型 | "该场地不是训练场地" |
| 6 | 车辆时间冲突 | "该车辆在此时段已被占用" |
| 7 | 场地容量超限 | "该场地在此时段已达到最大容纳车辆数(5)" |
| 8 | 教练时间冲突 | "您在此时段已有其他排班" |

> 提交成功后状态默认为 `0-待审核`，等待管理员审批。

```
PUT /schedules/{id}/audit
Content-Type: application/x-www-form-urlencoded

status=1&remark=同意
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| status | int | **是** | `1`=通过，`2`=拒绝 |
| remark | String | 否 | 审核备注（拒绝时建议填写原因） |

### 3.9 学员端 — 查看可约时段

```
GET /schedules/available?licenseType=C1
```

> **这是学员约课的第一步。** 系统自动查找学员绑定的教练，返回该教练所有 `status=1(已通过)` 且 `booked_count < max_students` 且 `start_time > 当前时间` 的排班。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| licenseType | String | 否 | 按车型筛选，不传返回全部的可用排班 |

**前置条件：**
- 学员必须已绑定教练（通过 `student_coach` 表关联）。如未绑定，返回错误 "您还没有绑定教练，请先申请分配教练"

**响应示例：**

```json
{
  "state": 20000,
  "data": [
    {
      "id": 5,
      "coachId": 1,
      "vehicleId": 1,
      "venueId": 4,
      "licenseType": "C1",
      "startTime": "2026-06-10T09:00:00",
      "endTime": "2026-06-10T11:00:00",
      "maxStudents": 2,
      "bookedCount": 1,
      "status": 1,
      "applyReason": "周三上午常规教学",
      "applyTime": "2026-06-05T10:00:00",
      "auditTime": "2026-06-05T11:00:00"
    }
  ]
}
```

> **前端提示**：`bookedCount < maxStudents` 的条件已由后端过滤，返回的都是有剩余名额的时段。展示时建议显示 `剩余名额 = maxStudents - bookedCount`。

## 四、约课管理（改动部分）

> 原有约课接口有重大变更，请仔细阅读下文。

### 4.1 约课状态变更

| status | 含义 | 说明 |
|--------|------|------|
| 0 | 待确认 | 学员提交，等待教练确认 |
| 1 | 已确认 | 教练已确认 |
| 2 | 已拒绝 | 教练拒绝（**注意：之前文档此处写"已完成"是错误的**） |
| 3 | 已取消 | 学员或管理员取消 |
| **4** | **已完成** | **新增状态**，教练确认上课完成 |

---

### 4.2 学员端 — 新增约课（重要变更）

```
POST /appointments
Content-Type: application/json

{
  "coachId": 1,
  "scheduleId": 5,
  "startTime": "2026-06-10T09:00:00",
  "endTime": "2026-06-10T10:00:00"
}
```

**请求体字段：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| coachId | int | **是** | 教练 coach_id |
| scheduleId | int | **是（推荐）** | 关联排班ID，传此字段后会校验排班归属+时段+名额 |
| startTime | datetime | **是** | 课程开始时间，必须在排班时段范围内 |
| endTime | datetime | **是** | 课程结束时间 |

**重要安全变更：**
- `studentId` **不再从请求体读取**，系统强制从 JWT Token 中获取当前登录学员的 userId，前端无需传递此字段
- 如果传了 `scheduleId`，系统自动校验排班归属、时间范围、剩余名额
- 如果没传 `scheduleId`，系统至少校验学员与教练的绑定关系

**系统校验流程：**

```
① 从 JWT 取 studentId（防伪造）
② 校验 startTime < endTime 且在未来
③ 若传了 scheduleId：
   ├─ 排班存在且 status=1
   ├─ 排班教练 = 学员绑定教练（student_coach 表）
   ├─ 约课时间在排班时段内
   └─ booked_count < max_students → 通过后 booked_count + 1
④ 若没传 scheduleId：
   └─ 校验 studentId→coachId 绑定关系存在
⑤ 校验学员自身时间无冲突
⑥ 保存约课（status=0 待确认）
```

---

### 4.3 学员端 — 查询我的约课（新增）

```
GET /appointments/my?page=1&size=10&status=1
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| page | int | 否 | 页码，默认 1 |
| size | int | 否 | 每页条数，默认 10 |
| status | int | 否 | 按状态筛选 |

> 自动从 JWT 取当前学员 userId，只返回该学员自己的约课。

---

### 4.4 学员端 — 取消约课（行为变更）

```
PUT /appointments/{id}/cancel?reason=临时有事
```

> **变更**：只能取消 `status=0(待确认)` 或 `status=1(已确认)` 的约课。已拒绝/已完成/已取消的不可取消。

> **新增行为**：取消约课时，如果该约课关联了排班（schedule_id 非空），系统会自动将排班的 `booked_count - 1`，释放名额。

---

### 4.5 教练端 — 确认约课

```
PUT /coach-portal/appointments/{id}/confirm
```

> 将约课状态从 `0(待确认)` → `1(已确认)`。只能操作自己的约课。

---

### 4.6 教练端 — 拒绝约课（行为修正）

```
PUT /coach-portal/appointments/{id}/reject
Content-Type: application/x-www-form-urlencoded

reason=该时段已有其他安排
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| reason | String | **是** | 拒绝原因 |

> **重要修正**：拒绝后状态为 `2(已拒绝)`，**不是** 3(已取消)。之前文档此处有误。

---

### 4.7 教练端 — 完成约课（新增）

```
PUT /coach-portal/appointments/{id}/complete
```

> 将已确认（status=1）的约课标记为 `4(已完成)`。只能操作自己的约课。

---

### 4.8 教练端 — 查看待确认约课

```
GET /coach-portal/appointments/pending
```

> 返回当前教练所有 `status=0(待确认)` 的约课。

---

### 4.9 教练端 — 查看约课日历

```
GET /coach-portal/appointments/calendar?startDate=2026-06-01&endDate=2026-06-30
```

> **修正**：现在返回 `status=1(已确认)` 或 `status=4(已完成)` 的约课。之前错误地包含了 `status=2(已拒绝)`。

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| startDate | String | 否 | 开始日期 `yyyy-MM-dd` |
| endDate | String | 否 | 结束日期 `yyyy-MM-dd` |

---

### 4.10 管理员端 — 查询所有约课（权限变更）

```
GET /appointments?page=1&size=10&studentId=4&coachId=1&status=0
```

> **变更**：此接口现在需要管理员权限（`@RequireRole(3)`）。之前任意登录用户都能查看所有约课。

---

### 4.11 管理员端 — 取消约课（新增）

```
PUT /appointments/{id}/admin-cancel?reason=学员违规操作
```

> 管理员可取消任何状态非终态的约课。如约课关联了排班，会自动释放名额。

## 五、教练空闲时间（逻辑变更）

### 5.1 查看空闲时间（返回内容变更）

```
GET /coach-portal/available-time
```

> **变更**：现在返回两部分数据：

```json
{
  "state": 20000,
  "data": {
    "manual": {
      "monday": ["09:00-12:00"],
      "wednesday": ["14:00-17:00"]
    },
    "schedules": [
      {
        "scheduleId": 5,
        "vehicleId": 1,
        "venueId": 4,
        "licenseType": "C1",
        "startTime": "2026-06-10T09:00:00",
        "endTime": "2026-06-10T11:00:00",
        "maxStudents": 2,
        "bookedCount": 1,
        "remaining": 1
      }
    ]
  }
}
```

| 字段 | 说明 |
|------|------|
| `manual` | 教练手动设置的 JSON 空闲时间（旧方式，保留） |
| `schedules` | **新增**，从已批准排班自动派生的可约时段，`remaining` 为剩余名额 |

> **前端建议**：优先展示 `schedules` 中的数据，因为这是有车辆和场地保障的时段。

## 六、场地管理（扩展字段）

### 6.1 场地新增字段

`GET /venues`、`GET /venues/{id}` 返回的场地数据中新增以下字段：

| 字段 | 类型 | 说明 | 适用类型 |
|------|------|------|---------|

**示例：**

```json
{
  "id": 4,
  "venueType": 2,
  "name": "南岸区训练基地",
  "address": "南岸区",
  "maxVehicles": 5,
  "supportedTypes": "C1,C2",
  "status": 1
}
```

## 八、前端实现建议

### 8.1 教练排班申请页面流程

```
1. 进入排班申请页面
2. 调用 GET /vehicles/available?vehicleType={我的准教车型}     → 获取可选车辆
3. 调用 GET /venues?venueType=2                                → 获取训练场地列表
4. 选择车辆、场地、时段、填写车型和最大容纳学员数
5. 提交 POST /coach-portal/schedules
6. 若返回 40900，说明有时间/场地冲突，提示教练更换时段
7. 提交成功后提示"已提交审核，请等待管理员审批"
```

### 8.2 学员约课页面流程

```
1. 进入约课页面
2. 调用 GET /schedules/available?licenseType={我的车型}       → 获取教练可约排班
3. 若无数据，提示"您的教练暂未发布可约时段"或"请先申请分配教练"
4. 展示排班列表，每项显示：时段、场地、车型、剩余名额
5. 学员选择排班 → 填写约课时间（必须在排班时段范围内）
6. 提交 POST /appointments（带上 scheduleId）
7. 若返回 40900，说明该时段已有其他约课或名额已满
```

### 8.3 管理员审核排班页面流程

```
1. 进入排班审核页面
2. 调用 GET /schedules/pending                              → 获取待审核列表
3. 展示每条排班的：教练姓名、车牌号、场地、时段、车型
4. 点击通过 → PUT /schedules/{id}/audit?status=1
5. 点击拒绝 → PUT /schedules/{id}/audit?status=2&remark=原因
```

### 8.4 约课状态展示建议

| status | 标签颜色 | 标签文字 |
|--------|---------|---------|
| 0 | 橙色 | 待确认 |
| 1 | 蓝色 | 已确认 |
| 2 | 红色 | 已拒绝 |
| 3 | 灰色 | 已取消 |
| 4 | 绿色 | 已完成 |
