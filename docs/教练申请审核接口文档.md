# 教练申请审核接口文档

## 通用说明

### 统一响应格式

```json
{
  "state": 20000,
  "message": null,
  "data": { ... }
}
```

### 业务状态码

| state | 含义 |
|-------|------|
| 20000 | 成功 |
| 40000 | 请求参数错误 |
| 40100 | 未登录或 token 过期 |
| 40300 | 权限不足 |
| 40400 | 资源不存在 |
| 40900 | 数据冲突（如已有待审核申请） |

### 认证方式

所有接口需在请求头中携带 Token：

```
Authorization: Bearer <token>
```

### 角色说明

| 值 | 角色 |
|----|------|
| 1 | 学员 |
| 2 | 教练 |
| 3 | 管理员 |

### 申请状态说明

| status | 含义 |
|--------|------|
| 0 | 待审核 |
| 1 | 已通过（已自动绑定教练） |
| 2 | 已拒绝 |

---

## 一、学员端接口

基础路径：`/coach-applications`

### 1.1 学员提交教练选择/更换申请

学员主动申请指定教练。如果学员已有教练，提交后视为"更换教练"申请，由管理员审核。

```
POST /coach-applications?studentId=4&coachId=2
```

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| studentId | int | 是 | 学员 user_id |
| coachId | int | 是 | 目标教练 coach_id |

#### 校验规则

| 场景 | 行为 |
|------|------|
| 学员已有待审核的申请 | 返回 40900 "您已有待审核的申请，请等待管理员处理" |
| 学员已绑定该教练 | 返回 40900 "您已绑定该教练，无需重复申请" |

#### 请求示例

```
POST /coach-applications?studentId=4&coachId=2
```

#### 成功响应

```json
{
  "state": 20000,
  "data": null
}
```

#### 错误示例

**409 — 已有待审核申请：**
```json
{
  "state": 40900,
  "message": "您已有待审核的申请，请等待管理员处理",
  "data": null
}
```

**409 — 已绑定该教练：**
```json
{
  "state": 40900,
  "message": "您已绑定该教练，无需重复申请",
  "data": null
}
```

---

### 1.2 查询本人的申请记录

学员查看自己提交过的所有申请记录，按申请时间倒序排列。

```
GET /coach-applications/student/{studentId}
```

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| studentId | int | 路径参数 | 学员 user_id |

#### 成功响应

```json
{
  "state": 20000,
  "data": [
    {
      "id": 1,
      "studentId": 4,
      "coachId": 2,
      "sourceCoachId": null,
      "transferReason": null,
      "status": 1,
      "applyTime": "2026-06-04T10:00:00",
      "auditTime": "2026-06-04T11:00:00",
      "auditReason": null
    },
    {
      "id": 2,
      "studentId": 4,
      "coachId": 1,
      "sourceCoachId": null,
      "transferReason": null,
      "status": 0,
      "applyTime": "2026-06-05T10:00:00",
      "auditTime": null,
      "auditReason": null
    }
  ]
}
```

#### 返回字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 申请ID |
| studentId | int | 学员 user_id |
| coachId | int | 目标教练 coach_id |
| sourceCoachId | int | 发起移交的教练ID（学员申请时为 null） |
| transferReason | String | 教练移交原因（学员申请时为 null） |
| status | int | 0-待审核, 1-已通过, 2-已拒绝 |
| applyTime | datetime | 申请时间 |
| auditTime | datetime | 审核时间（待审核时为 null） |
| auditReason | String | 拒绝原因（通过或待审核时为 null） |

#### 前端展示建议

根据 `status` 显示不同状态的标签：

| status | 显示 |
|--------|------|
| 0 | 待审核（灰色/黄色标签） |
| 1 | 已通过（绿色标签，提示"已绑定教练"） |
| 2 | 已拒绝（红色标签，显示 `auditReason`） |

## 二、管理员端接口

基础路径：`/coach-applications`

### 2.1 查询所有待审核的申请

管理员查看所有状态为"待审核"的申请记录（含学员申请和教练移交），按申请时间倒序排列。

```
GET /coach-applications/pending
```

#### 请求参数

无

#### 成功响应

```json
{
  "state": 20000,
  "data": [
    {
      "id": 1,
      "studentId": 4,
      "coachId": 2,
      "sourceCoachId": null,
      "transferReason": null,
      "status": 0,
      "applyTime": "2026-06-05T10:00:00",
      "studentName": "王小明",
      "coachName": "李教练",
      "sourceCoachName": null,
      "applyType": "学员申请"
    },
    {
      "id": 2,
      "studentId": 5,
      "coachId": 1,
      "sourceCoachId": 2,
      "transferReason": "因身体原因无法继续带教",
      "status": 0,
      "applyTime": "2026-06-05T11:00:00",
      "studentName": "李芳",
      "coachName": "张教练",
      "sourceCoachName": "李教练",
      "applyType": "教练移交"
    }
  ]
}
```

#### 返回字段说明

| 字段 | 类型 | 说明 |
|------|------|------|
| id | int | 申请ID |
| studentId | int | 学员 user_id |
| coachId | int | 目标教练 coach_id |
| sourceCoachId | int | 发起移交的教练ID（null=学员申请） |
| transferReason | String | 教练移交原因（学员申请时为 null） |
| status | int | 0-待审核 |
| applyTime | datetime | 申请时间 |
| studentName | String | 学员真实姓名 |
| coachName | String | 目标教练真实姓名 |
| sourceCoachName | String | 发起移交的教练姓名（null=学员申请） |
| applyType | String | 申请类型："学员申请" 或 "教练移交" |

#### 前端展示建议

根据 `applyType` 区分展示样式：

| applyType | 展示建议 |
|-----------|---------|
| 学员申请 | 显示"学员申请"标签，操作区显示学员姓名 → 目标教练 |
| 教练移交 | 显示"教练移交"标签，操作区显示源教练 → 学员 → 目标教练 和移交原因 |

### 2.2 审核申请

管理员审核学员申请或教练移交申请。此接口同时服务于两种场景。

**通过时**：自动解绑旧教练 → 绑定新教练。
**拒绝时**：填写拒绝原因，原绑定关系不变。

```
PUT /coach-applications/{id}/audit?pass=true
```

#### 请求参数

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| id | int | 路径参数 | 申请ID |
| pass | boolean | 是 | true=通过, false=拒绝 |
| reason | String | 否 | 拒绝原因（pass=false 时建议填写） |

#### 请求示例

**通过申请：**
```
PUT /coach-applications/1/audit?pass=true
```

**拒绝申请：**
```
PUT /coach-applications/1/audit?pass=false&reason=该教练当前学员已满
```

#### 成功响应

```json
{
  "state": 20000,
  "data": null
}
```

#### 审核通过时的后台操作

1. **校验**：如果是教练移交申请，检查发起移交的教练是否仍与该学员有绑定关系
2. **解绑旧教练**：将该学员所有 `status=1` 的绑定置为 `status=0`
3. **绑定新教练**：创建新的 `student_coach` 记录，`status=1`

#### 校验规则

| 场景 | 行为 |
|------|------|
| 申请不存在 | 返回 40400 "申请记录不存在" |
| 教练移交但源教练已无绑定 | 返回 40900 "该学员已不是发起移交的教练名下的学员，请核实" |

#### 错误示例

**404 — 申请不存在：**
```json
{
  "state": 40400,
  "message": "申请记录不存在",
  "data": null
}
```

**409 — 移交场景源教练已无绑定：**
```json
{
  "state": 40900,
  "message": "该学员已不是发起移交的教练名下的学员，请核实",
  "data": null
}
```

## 三、业务流程

### 学员申请更换教练完整流程

```
学员端                           管理员端
  │                                │
  ├─ 提交教练选择/更换申请           │
  │  POST /coach-applications       │
  │  ?studentId=4&coachId=2        │
  │                                │
  │                    ┌────────────┤
  │                    │ 查询待审核列表 │
  │                    │ GET /coach-  │
  │                    │ applications/ │
  │                    │ pending      │
  │                    ├────────────┤
  │                    │ 审核申请     │
  │                    │ PUT /coach-  │
  │                    │ applications/ │
  │                    │ {id}/audit   │
  │                    │ ?pass=true   │
  │                    │             │
  │                    │ 通过时自动：  │
  │                    │ 解绑旧教练 →  │
  │                    │ 绑定新教练    │
  │                    └──────┬─────┘
  │                           │
  ├─ 查看申请结果               │
  │  GET /coach-applications/  │
  │  student/{studentId}       │
  │  → 看到 status 变为 1 或 2  │
  │                           │
  │  如已通过：查看我的教练已更新   │
  │  GET /coach-assignments/    │
  │  my-coach/{studentId}       │
```

## 四、接口路径速查

| 操作 | 方法 | 路径 | 角色 |
|------|------|------|------|
| 学员提交申请 | POST | `/coach-applications` | 学员 |
| 查询本人的申请记录 | GET | `/coach-applications/student/{studentId}` | 全部登录 |
| 查询待审核列表 | GET | `/coach-applications/pending` | 管理员 |
| 审核申请 | PUT | `/coach-applications/{id}/audit` | 管理员 |

## 五、前端对接建议

### 学员端页面

**申请更换教练页：**
- 搜索目标教练
- 调 `GET /coach-assignments/recommend` 获取推荐列表
- 点击"申请"按钮 → 调 `POST /coach-applications`
- 成功后提示 "申请已提交，等待管理员审核"
- 防重复提交：按钮置灰处理

**申请记录页：**
- 调 `GET /coach-applications/student/{studentId}` 展示列表
- 用 `status` 字段区分显示状态标签
- 已拒绝的显示 `auditReason`

### 管理员端页面

**待审核列表页：**
- 调 `GET /coach-applications/pending` 展示
- 通过 `applyType` 区分"学员申请"和"教练移交"
- 学员申请：显示 "学员XXX 申请 教练XXX"
- 教练移交：显示 "教练XXX 申请将 学员XXX 移交给 教练XXX（原因：...）"
- 提供"通过"和"拒绝"操作按钮

**审核操作：**
- 通过：直接调 `PUT /coach-applications/{id}/audit?pass=true`，成功后刷新列表
- 拒绝：弹窗要求填写拒绝原因，调 `PUT /coach-applications/{id}/audit?pass=false&reason=原因`
